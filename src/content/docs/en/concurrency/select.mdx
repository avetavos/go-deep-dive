## The select statement

`select` is Go's way to wait on multiple channel operations simultaneously. It blocks until one of its cases is ready, then executes that case. If multiple cases are ready at the same time, one is chosen at **random** — this is a deliberate language design choice to avoid starvation.

```go
select {
case v := <-ch1:
    fmt.Println("received from ch1:", v)
case ch2 <- 99:
    fmt.Println("sent to ch2")
}
```

A `select` case can be a send or a receive — whichever operation can proceed without blocking.

## The default clause

Adding a `default` case makes the `select` non-blocking. If no channel is ready, the `default` branch runs immediately:

```go
select {
case v := <-ch:
    fmt.Println("got", v)
default:
    fmt.Println("nothing ready yet")
}
```

This is the idiomatic way to poll a channel without blocking.

## Timeouts

A common pattern is combining `select` with `time.After`, which returns a channel that receives a value after the specified duration:

```go
select {
case result := <-work:
    fmt.Println("done:", result)
case <-time.After(2 * time.Second):
    fmt.Println("timed out")
}
```

For production code where many timeouts are set, prefer `context.WithTimeout` (covered in the Context lesson) because it propagates cancellation and avoids timer leaks.

## Runnable example

The snippet below demonstrates all three select behaviours in sequence — deterministically by using separate selects with pre-loaded buffered channels, followed by a `default` branch demo.

export const selectCode = `package main

import "fmt"

func main() {
\t// Pre-load two buffered channels so both are ready
\tch1 := make(chan string, 1)
\tch2 := make(chan string, 1)
\tch1 <- "one"
\tch2 <- "two"

\t// Drain ch1 with its own select — guaranteed to receive "one"
\tselect {
\tcase msg := <-ch1:
\t\tfmt.Println("ch1:", msg)
\t}

\t// Drain ch2 with its own select — guaranteed to receive "two"
\tselect {
\tcase msg := <-ch2:
\t\tfmt.Println("ch2:", msg)
\t}

\t// default makes select non-blocking
\tch3 := make(chan int)
\tselect {
\tcase v := <-ch3:
\t\tfmt.Println("received", v)
\tdefault:
\t\tfmt.Println("no value ready, default taken")
\t}

\t// Fan-in: select from whichever of two channels has a value
\t// (use separate selects for deterministic playground output)
\tr1 := make(chan int, 1)
\tr2 := make(chan int, 1)
\tr1 <- 10
\tr2 <- 20
\tselect {
\tcase v := <-r1:
\t\tfmt.Println("r1:", v)
\t}
\tselect {
\tcase v := <-r2:
\t\tfmt.Println("r2:", v)
\t}

\tfmt.Println("done")
}`;

<Callout title="Gotcha: empty select blocks forever">
A `select` with no cases — `select {}` — blocks the current goroutine forever. This is occasionally used intentionally to keep `main` alive (a leaner alternative to `for {}`), but it is more commonly a mistake. Always ensure your select has at least one reachable case or a `default`. A `select` with only a `default` and no channel cases executes the `default` immediately every time — useful for a non-blocking poll, but an infinite loop if placed inside `for` without additional logic.
</Callout>
